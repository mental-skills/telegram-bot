# Задание для Codex: MVP Telegram-бота Mental Skills

Создай рабочий MVP Telegram-бота — образовательного тренажёра решений для родителей юных футболистов.

## Источники требований

1. `docs/Mental_Skills_TZ_Telegram_Bot_MVP_branded_v2.docx` — основной контракт разработки.
2. `content/scenario.schema.json` — обязательный формат контента.
3. `content/PREMATCH_INSTRUCTIONS_02.json` — первая тестовая ситуация.
4. `docs/Mental_Skills_football_parent_trainer_revised.docx` — методическая база; не переносить её длинные тексты в код автоматически.
5. `assets/asset_manifest.json` — обязательный каталог визуальных материалов.
6. `assets/reference/` — два главных бренд-референса; использовать при разработке интерфейса, но не отправлять пользователю.
7. `assets/runtime/` — готовые изображения для экранов Telegram-бота.
8. `assets/brand_tokens.json` — обязательные цвета, композиционные принципы и правила доступности.
9. `assets/visual_usage_map.json` — карта применения ассетов по ролям экранов.
10. `docs/VISUAL_GUIDE.md` — краткая инструкция по реализации визуального слоя.

## Первая итерация

Реализуй только инфраструктуру и сценарий `PREMATCH_INSTRUCTIONS_02`.

### Функции

1. `/start` и главное меню.
2. Информационный экран о том, что это образовательный тренажёр, а не диагностика.
3. Выбор возраста ребёнка: 6–8, 9–12, 13–16 лет.
4. Загрузка и валидация сценария из JSON при запуске приложения.
5. Поддержка узлов: `info`, `choice`, `outcome`, `advice`, `tool`, `completion`, `emergency`.
6. Переходы по кнопкам без хранения пользовательских текстов в Python-коде.
7. Сохранение прогресса: пользователь, возраст, сценарий, текущий узел, история выборов, завершённость.
8. Продолжение незавершённой ситуации.
9. Повтор ситуации и возврат в меню.
10. Сохранение скрытых оценок и сигналов только в базе; никогда не показывать их пользователю.
11. Короткая словесная обратная связь без процентов и диагностических ярлыков.
12. Корректная обработка устаревшей кнопки и повторного нажатия.
13. Логирование технических событий без сохранения текста личных сообщений.
14. Команда `/privacy` с кратким описанием обрабатываемых данных.
15. Команда `/reset` с подтверждением удаления прогресса.


## Визуальный слой Mental Skills — обязателен

Оформи бот в фирменной стилистике Mental Skills, используя только изображения из каталога `assets/`.

### Источники визуального языка

- `assets/reference/brand_graphic_elements.png` — главный референс фирменных линий, рамок, паттернов, акцентов и инфографики.
- `assets/reference/brand_illustration_style.png` — главный референс света, динамики, спортивной среды и общего уровня качества.

Эти два изображения служат **только дизайн-референсами**. Не отправляй их пользователям в Telegram.

### Готовые runtime-ассеты

- `brand_logo_horizontal` — `/start`, «О проекте», нейтральная бренд-заставка;
- `ui_dialogue_choice` — вступление в ситуацию и экран выбора;
- `ui_question_card` — FAQ и поясняющий вопрос;
- `ui_achievement` — завершение ситуации или модуля;
- `ui_practice_task` — практический совет и упражнение;
- `ui_location` — будущий экран адреса/записи;
- `ui_insight` — главная мысль и общий вывод;
- `ui_neural_focus` — ритуал переключения и ментальный инструмент;
- `ui_rating` — словесная итоговая обратная связь без процентов;
- `ui_calendar` — практика до следующей игры или напоминание;
- `ui_tool_cards` — библиотека практических инструментов.

Точные пути, назначения и текстовые описания находятся в `assets/asset_manifest.json`.


### Точное применение приложенных брендбордов

- Считай `brand_illustration_style.png` эталоном премиального спортивного реализма, динамики, фокуса, естественных пропорций и наставнического взаимодействия.
- Считай `brand_graphic_elements.png` эталоном нейронных линий, жёлтых рамок, технологических паттернов, чёрно-графитового пространства и акцентов.
- Сохраняй исходные пропорции `brand_logo_horizontal.png`; не обрезай знак мозга и надпись.
- Не копируй мелкий текст из брендбордов в интерфейс. Извлекай только визуальные принципы.
- Для обычного Telegram MVP используй только готовые runtime-карточки. Брендборды и tokens готовят архитектуру к будущему Mini App.

### Реализация медиа

1. Добавь `AssetRepository`, который загружает и валидирует `asset_manifest.json`, `brand_tokens.json` и `visual_usage_map.json`, проверяет наличие файлов и возвращает путь, назначение, описание, размеры и alt-текст ассета.
2. Поддержи необязательное поле `media` в JSON-узле сценария. Используй `scenario.schema.json` версии 1.2, включая `presentation` и `caption_strategy`.
3. `ScenarioEngine` должен возвращать медиа вместе с текстом и кнопками, не зная физических путей файлов.
4. Telegram-рендерер должен:
   - использовать `send_photo` с подписью и inline-кнопками, когда подпись помещается в лимит Telegram;
   - при длинном тексте сначала отправлять изображение, затем отдельное текстовое сообщение с кнопками;
   - при отсутствии или повреждении файла логировать предупреждение и показывать полноценный текстовый экран;
   - не ломать переход и не терять прогресс из-за ошибки изображения.
5. Не отправляй изображение на каждом узле. Используй визуальную карточку только на смысловых этапах: вступление, общий вывод, практический инструмент, домашнее упражнение и завершение.
6. Не встраивай динамический текст в изображения. Весь смысл, варианты и советы должны оставаться в сообщении Telegram и быть понятными без картинки.
7. Не пытайся перекрашивать системные кнопки Telegram: фирменный стиль передаётся изображениями, структурой текста и единообразной подачей.
8. Не изменяй, не перерисовывай и не заменяй предоставленные изображения без отдельного согласования.
9. При первом отправлении локального файла можно использовать `FSInputFile`; после получения Telegram `file_id` допускается кешировать его в памяти процесса. Архитектура должна позволять позднее добавить постоянный кеш.
10. Для `/start` используй `brand_logo_horizontal`, для библиотеки инструментов — `ui_tool_cards`, для финального словесного итога будущего полного модуля — `ui_rating`.

### Визуальные тесты и проверки

Добавь тесты, подтверждающие:

- все `asset_id` из сценария существуют в manifest;
- все runtime-файлы существуют;
- design-reference ассеты не могут быть отправлены пользователю;
- отсутствие картинки приводит к text-only fallback;
- поля `media` и внутренние пути не выводятся пользователю как текст;
- сценарий №2 использует визуальные карточки только на предусмотренных смысловых этапах.

## Архитектурные требования

- Тексты сценариев хранятся только в JSON.
- Слой Telegram не должен знать структуру конкретной ситуации.
- Создай универсальный `ScenarioEngine`, который получает сценарий, возраст и текущий узел и возвращает экран и кнопки.
- Добавь `ContentRepository` с кешированием и проверкой JSON Schema.
- Используй repository/service pattern без избыточной сложности.
- Callback data должна быть короткой и укладываться в ограничение Telegram 64 байта; в ней не нужно передавать полный ID узла.
- Защити переходы от двойного нажатия и перехода по кнопке из старого сообщения.
- Данные одного пользователя не должны смешиваться с данными другого.
- Все секреты только в `.env`; создай `.env.example` без настоящих значений.

## База данных

Минимальные таблицы:

- `users`;
- `trainer_sessions`;
- `choice_events`;
- `content_versions`.

В `choice_events` сохранить код перехода, ID сценария, ID финальной ветки и скрытые сигналы. Не сохранять полный текст сообщений пользователя.
